Why Interactive?

Welcome to Workshop 2! Yesterday you built great static visualizations. Today we take those same skills and make them interactive. The big question is: why bother? Static plots are fine for a PDF report, but your DSPG final deliverable is usually a dashboard that community partners and stakeholders will click through. Interactivity lets them explore the data on their own terms, without needing you in the room. Let’s walk through the three main approaches we’ll cover today.

Static vs. Interactive

Static plots are great for:

  • Papers, PDF reports, printed posters
  • When the story is fully told by one view

Interactive plots add value when:

  • Stakeholders want to explore the data themselves
  • There are many subgroups, filters, or geographic areas
  • Your deliverable lives on the web (HTML, dashboard)

DSPG final deliverables are often community dashboards — interactivity is not a bonus, it is the product.

DSPG Use Cases

Your project probably has some of these:

  • County-level data → interactive choropleth map
  • Data by group → bar chart that updates when you change a filter
  • Trend over time by group → hover-to-see-value line chart
  • Multiple variables → linked scatter + map + table

All of these are achievable today with the packages we will cover.

Mention a few real DSPG project types if you know them — rural broadband, food access, workforce data. The key message: every one of those can produce a great interactive deliverable with the tools in this workshop.

Two Tracks to Interactivity

Shiny Apps HTML Widgets
How it works R keeps running; browser sends input → R re-computes → result returned R runs once to produce HTML; JavaScript handles all interactivity in the browser
Pros Full reactivity; server-side logic for large data / databases; complete control No server needed; share as a static file; embeds in Quarto / GitHub Pages; less setup
Cons Requires a running R process (local or hosted); more setup code All data lives in the browser — not for large datasets; no server-side re-computation
Use this when Filters, models, or outputs must recompute on the fly You need hover / zoom / linked views and your data fits in a browser

The key insight: if your deliverable is a static HTML file (e.g., hosted on GitHub Pages), you can still have impressive interactivity using widgets and crosstalk. Shiny requires a server.

Part 1: Shiny Apps

The Three Parts of Every Shiny App

library(shiny)

# 1. UI: what the user sees and interacts with
ui <- fluidPage(
  titlePanel("My App"),
  sidebarLayout(
    sidebarPanel(
      # inputs go here
    ),
    mainPanel(
      # outputs go here
    )
  )
)

# 2. Server: the logic that produces outputs from inputs
server <- function(input, output, session) {
  # reactive logic goes here
}

# 3. Launch the app
shinyApp(ui = ui, server = server)

Layout Functions — Syntax

Function What it creates
fluidPage(...) Responsive full-width page — the default container
sidebarLayout(sidebarPanel(...), mainPanel(...)) Classic two-column: controls left, output right
fluidRow(column(width, ...), ...) Free grid — place anything in columns 1–12
navbarPage("Title", tabPanel(...), ...) Multi-tab app with a top navigation bar
fillPage(...) Full-viewport page — good for maps and dashboards

Start with sidebarLayout for any app with filters on one side and a chart on the other. Upgrade to navbarPage when you need multiple views.

  tabPanel("Sidebar",
    sidebarLayout(
      sidebarPanel(
        sliderInput("bins", "Number of bins:", min = 5, max = 50, value = 20)
      ),
      mainPanel(
        plotOutput("hist")
      )
    )
  ),

  # Tab 2 — fluidRow + column: flexible grid layout
  tabPanel("Grid",
    fluidRow(
      column(4, wellPanel(
        selectInput("var", "Variable:", choices = c("mpg", "hp", "wt"))

      column(8,
        plotOutput("scatter")
      )
    )

server <- function(input, output, session) {
  output$hist <- renderPlot({
    hist(mtcars$mpg, breaks = input$bins,
         col = "#861F41", border = "white", main = "MPG Distribution")
  })
  output$scatter <- renderPlot({
    plot(mtcars[[input$var]], mtcars$mpg,
         xlab = input$var, ylab = "mpg", col = "#E5751F", pch = 16)
  })
  output$full_plot <- renderPlot({
    par(mar = c(4, 4, 2, 1))
    plot(mtcars$wt, mtcars$mpg, col = "#861F41", pch = 16,
         xlab = "Weight", ylab = "MPG", main = "Weight vs MPG")
  })
}

Output Functions — Syntax

Every output is a pair: one function in the UI (where to display it), one in the server (how to compute it).

UI function Server function Renders
plotOutput("id") renderPlot({}) Static ggplot2 chart
plotlyOutput("id") renderPlotly({}) Interactive plotly chart
leafletOutput("id") renderLeaflet({}) Interactive leaflet map
DTOutput("id") renderDT({}) Interactive DT table
tableOutput("id") renderTable({}) Simple HTML table
textOutput("id") renderText({}) Text string
verbatimTextOutput("id") renderPrint({}) Console-style output

The id string must match exactly between the UI and the server: plotOutput("scatter") ↔︎ output$scatter.

Output Functions — Swapping Static for Interactive

The only two lines that change when you swap plotly or leaflet into a Shiny app:

Server: reactive() — Shared Intermediate Values

When multiple outputs need the same filtered or transformed data, compute it once with reactive() and reuse it everywhere.

server <- function(input, output, session) {

  # reactive() runs whenever input$species changes,
  # and caches the result for all outputs that need it
  filtered <- reactive({
    penguins |>
      filter(species == input$species) |>
      drop_na()
  })

  # Both outputs call filtered() — no duplicated filter code
  output$scatter <- renderPlot({
    ggplot(filtered(), aes(bill_length_mm, body_mass_g)) +
      geom_point(color = "#861F41") + theme_minimal(base_size = 13)
  })

  output$summary <- renderTable({
    filtered() |> summarise(n = n(), mean_mass = mean(body_mass_g))
  })
}

Call a reactive object with () — filtered() not filtered. The result is cached: if two outputs call it in the same update cycle, the filter runs only once.

  mutate(county = str_remove(NAME, ", Virginia"))

ui <- fluidPage(
  titlePanel("Virginia County Explorer"),
  sidebarLayout(
    sidebarPanel(
      selectInput("county", "County:", choices = sort(unique(va$county)),
                  selected = "Fairfax County"),
      selectInput("var", "Variable:", choices = names(vars))
    ),
    mainPanel(
      textOutput("headline"),
      leafletOutput("map", height = 340),
      tableOutput("tbl")
    )
  )

server <- function(input, output, session) {

  # reactive: filter to selected variable — shared by all three outputs
  var_data <- reactive({
    va |> filter(variable == input$var)   # tidycensus stores names, not codes
  })

  output$headline <- renderText({
    row       <- var_data() |> filter(county == input$county)
    state_med <- median(var_data()$estimate, na.rm = TRUE)
    dir       <- if (row$estimate >= state_med) "above" else "below"
    paste0(input$county, " — ", input$var, ": ", comma(round(row$estimate)),
           " (", dir, " the VA median of ", comma(round(state_med)), ")")
  })

  output$map <- renderLeaflet({
    df  <- var_data()
    pal <- colorNumeric("YlOrRd", df$estimate, na.color = "grey80")
    leaflet(df) |> addTiles() |>
      addPolygons(fillColor = ~pal(estimate), fillOpacity = 0.7,
                  color = "white", weight = 1,
                  popup = ~paste0("<b>", county, "</b><br>",
                                  input$var, ": ", comma(estimate))) |>
      addPolygons(data = filter(df, county == input$county),
                  fill = FALSE, color = "#861F41", weight = 3) |>
      addLegend(pal = pal, values = ~estimate,
                title = input$var, position = "bottomright")
  })

  output$tbl <- renderTable({
    d <- var_data()
    tibble(Metric = c(input$county, "VA Median", "VA Min", "VA Max"),
           Value  = comma(round(c(filter(d, county == input$county)$estimate,
                                  median(d$estimate, na.rm = TRUE),
                                  min(d$estimate,    na.rm = TRUE),
                                  max(d$estimate,    na.rm = TRUE)))))
  }, striped = TRUE, hover = TRUE, bordered = TRUE)
}

Leaflet with Multiple Layers

Leaflet Multi-Layer: The Pattern

Add each layer as a named group, then use leafletProxy() to show or hide groups without redrawing the whole map.

# UI: checkbox drives which layers are visible
checkboxGroupInput("layers", "Show layers:",
                   choices  = c("Income", "Poverty Rate", "Median Age"),
                   selected = "Income")

# Server: build the map once with all layers, two hidden by default
output$map <- renderLeaflet({
  leaflet(va) |>
    addTiles() |>
    addPolygons(group = "Income",       ...) |>
    addPolygons(group = "Poverty Rate", ...) |>
    addPolygons(group = "Median Age",   ...) |>
    hideGroup("Poverty Rate") |>
    hideGroup("Median Age")
})

# observe() reacts to checkbox changes and toggles groups
observe({
  proxy <- leafletProxy("map", session)
  for (layer in c("Income", "Poverty Rate", "Median Age")) {
    if (layer %in% input$layers) proxy |> showGroup(layer)
    else                         proxy |> hideGroup(layer)
  }
  # Rebuild legends to match visible layers
  proxy |> clearControls()
  if ("Income" %in% input$layers)
    proxy |> addLegend("bottomleft", pal = pal_income, values = va$income,
                       title = "Median Income ($)")
})

leafletProxy() updates an existing map in-place — the viewport and zoom level are preserved. Use it for any server-side map update after first render.

Part 2: HTML Widgets

HTML Widgets run entirely in the browser — R produces the HTML once, and JavaScript handles all interactivity. No server needed. You can share the output as a standalone .html file, embed it in a Quarto report, or host it on GitHub Pages.

Interactive Figures with plotly

plotly is probably the easiest on-ramp to interactivity. If you have a ggplot2 chart you are proud of, one function call — ggplotly() — turns it interactive. Let’s see how.

ggplotly() — One Line to Interactive

Take any ggplot2 chart and wrap it in ggplotly():

library(plotly)

p <- penguins |>
  drop_na() |>
  ggplot(aes(x = bill_length_mm, y = body_mass_g,
             color = species, shape = species)) +
  geom_point(alpha = 0.7, size = 2) +
  labs(title = "Penguin Bill Length vs Body Mass",
       x = "Bill Length (mm)", y = "Body Mass (g)") +
  theme_minimal(base_size = 13)

Just by calling ggplotly(), users can:

No extra code required.

Saving and Sharing HTML Widgets

Any widget (plotly, leaflet, DT, crosstalk) can be saved as a standalone HTML file:

library(htmlwidgets)

# Save any widget to a self-contained HTML file
p <- ggplotly(my_ggplot)
saveWidget(p, "my_chart.html", selfcontained = TRUE)

m <- leaflet(data) |> addTiles() |> addCircleMarkers(...)
saveWidget(m, "my_map.html", selfcontained = TRUE)

Ways to share:

  • Email the .html file directly — opens in any browser, no R needed
  • Google Drive / Dropbox — upload and share the link
  • Quarto report — embed widgets directly; render .qmd → HTML and they are included automatically
  • GitHub Pages — commit the HTML file and share the public URL

selfcontained = TRUE bundles all JavaScript into one file (~5–20 MB). For lighter files, use selfcontained = FALSE — R creates a companion _files/ folder alongside the HTML; share both together.

Linked Widgets with crosstalk

crosstalk lets you link a table, a chart, and a map so that filtering one automatically filters the others — and you do not need a Shiny server at all. The output is a static HTML file you can post on GitHub Pages or email to a stakeholder.

The crosstalk Idea

One SharedData object → all widgets see the same selection state.

library(crosstalk)

shared <- SharedData$new(my_data)

# Now pass `shared` instead of `my_data` to each widget
plot_ly(shared, ...)
datatable(shared, ...)
leaflet(shared) |> addCircleMarkers(...)

Filtering in any one widget instantly updates the others — no server, no Shiny.

Crosstalk Demo: Chart + Table

cities <- tibble(
  city       = c("New York", "Los Angeles", "Chicago", "Houston",
                 "Phoenix", "Philadelphia", "San Antonio", "San Diego",
                 "Dallas", "San Jose"),
  state      = c("NY", "CA", "IL", "TX", "AZ", "PA", "TX", "CA", "TX", "CA"),
  lat        = c(40.71, 34.05, 41.88, 29.76, 33.45,
                 39.95, 29.42, 32.72, 32.78, 37.34),
  lng        = c(-74.01, -118.24, -87.63, -95.37, -112.07,
                 -75.16, -98.49, -117.16, -96.80, -121.89),

When to Use crosstalk

Use crosstalk when:

  • Your deliverable is a static HTML file (no server)
  • You want linked filtering across 2–3 widgets
  • The dataset is small to medium (all data goes to the browser)

Reach for Shiny when:

  • You need server-side filtering (large data)
  • You need dynamic data loading or database queries
  • You need complex conditional logic (observeEvent, eventReactive)

Key Takeaways

What We Covered Today

Part 1 — Shiny Apps (server required):

  • ui + server + shinyApp() — the minimal skeleton
  • bslib — modern Bootstrap 5 themes with one function call
  • renderPlotly() / renderLeaflet() — drop any widget into Shiny
  • Reactivity: input$x triggers re-computation automatically

Part 2 — HTML Widgets (no server):

  • plotly — ggplotly() turns any ggplot interactive in one line
  • ggiraph — interactive ggplot2 geoms with drop-in replacements
  • DT — sortable, filterable, downloadable data tables
  • leaflet + tigris — interactive maps with Census-pulled boundaries
  • crosstalk — SharedData$new() links all of the above, no server needed

The Decision Tree

Does your deliverable need server-side logic
(large data, database, complex conditions)?
    │
    ├─ YES → Build a Shiny app (Part 1)
    │
    └─ NO  → Do you need widgets to talk to each other?
                 │
                 ├─ YES → Use crosstalk (SharedData)
                 │
                 └─ NO  → Use standalone widgets
                          (plotly / leaflet / DT / ggiraph)
